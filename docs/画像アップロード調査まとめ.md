# SalonBoard スタイル/フォトギャラリー画像アップロード 失敗調査まとめ

予約同期くん（worker / Playwright）経由で、SalonBoard の **美容室スタイル登録 (/CNB styleEdit)** と
**エステ フォトギャラリー (/CNK photoGalleryEdit)** の **画像アップロード**が失敗し続けている件の、
これまでの調査・対策・原因の可能性の全記録。

期間: 2026-06-08 〜 2026-06-09 / バージョン: v0.2.110 〜 v0.2.135

---

## 1. 症状

- 画像アップロードの**ローディングがずっと終わらず**、最終的に **「通信に失敗しました」**（SalonBoard のダイアログ）で失敗。
- **手動**（人間が普段使いの Chrome で操作）だと**一瞬で成功**する。
- **予約同期くん（自動）経由でだけ失敗**する。
- **スクレイピングで開いた Chrome を人間が手動で操作してもローディングが終わらない** ← 最重要の手がかり。
- ブログのカバー画像アップロードは別モーダル方式で**成功している**。

---

## 2. アップロードの仕組み（実JS / 実DOM 解析で確定）

美容室・エステとも共通の `CN_CMN_imageUploaderModal.js` を使う。**2段 AJAX**：

1. **モーダルを開く**: アップロードUI（美容室=`img#FRONT_IMG_ID_IMG` / エステ=`a.jscUploadImg`・`img.mod_btn_upload`）を
   **jQuery 委譲ハンドラ**でクリック → `img_upload_modal_view()` が `/CNB(or CNK)/imgreg/imgUpload/` を AJAX で叩き、
   モーダルHTML（`#imgUploadForm` 入り）を `#imageUploaderModalBody` に描画。グローバル `CONTEXT_URL_STR` / `URL_STR` をセット。
2. **登録する**: `input.jscImageUploaderModalSubmitButton`（value="登録する"）クリック → `file_upload()`:
   - ファイルは `input.jscImageUploaderModalInput` の `change` で `prepareFileInfo()` → グローバル `window.waitImgeFile` に保持。
   - `FormData.append('formFile', window.waitImgeFile)` ＋ `#imgUploadForm` の hidden
     (`setImgId`/`dataKey`/`targetActionId`/TOKEN/`STORE_ID`/`modified`/`pubManageId`/`wFlg`) を
     `/imgreg/imgUpload/doUpload?wFlg=...` へ `$.ajax`（**timeout 35秒**）。
   - 応答HTMLの `#imageId`/`#elementName`/`#imageFilePath` を `setUploadImage()` で親フォーム `FRONT_IMG_ID` 等へ反映。

参照DOM:
- `salonboard_code/美容室/スタイル登録_styleEdit.html`
- `salonboard_code/美容室/スタイル画像アップロードモーダル_imageUploaderModal.html`
- `salonboard_code/エステサロン/フォトギャラリー_photoGalleryEdit.html`

## 3. 途中で見つけて潰したバグ（いずれも真因ではないが必須の修正）

| # | 症状 | 原因 | 修正(版) |
|---|------|------|---------|
| A | `imageId_not_set`（waitImgeFile空でformFile空→35sタイムアウト） | ファイルを generic な `input[type=file]` にセットしていて、正しい `input.jscImageUploaderModalInput` の change が走らず `waitImgeFile` が空 | 正しい input に setInputFiles + change 発火 (v0.2.114) |
| B | 「登録するが押されず画面が消える」 | 登録ボタンを `a:has-text` / `input[type=submit]` で探していたが、実体は `input[type=button].jscImageUploaderModalSubmitButton` | 正しいセレクタで1回だけクリック (v0.2.112〜113) |
| C | 直接POSTが一度も実行されず常にブラウザXHRに落ちる (`params=null`) | パラメータ取得 evaluate が `#imgUploadForm input:hidden[name=..]` という **jQuery専用 `:hidden`** を querySelector に渡し SyntaxError → evaluate全体が reject | 有効なCSSに修正 (v0.2.124) |
| D | エステ `no_file_input`（モーダルが開かない） | アップロードUIのクリックが**普通の click だと jQuery 委譲ハンドラが走らず**モーダルが開かない | 本物の MouseEvent(mousedown/mouseup/click) を dispatch (v0.2.127 エステ / v0.2.128 美容室) |
| E | デバッグキャプチャが書けず原因不明 | `captureScrapeDebug` の `page.content()`/`screenshot()` がローディング中にハングしフォルダごと残らない | meta.json 最優先 + content/screenshot を timeout 化 (v0.2.119) |
| F | 直接POST成功(HTTP200)後に二重送信でERR_ABORTED | 200後にブラウザXHRにもフォールバックして二重送信 | 200後はブラウザXHRに落とさず imageId を直接 FRONT_IMG_ID に反映 (v0.2.129) |

## 4. 試した送信方式と結果

| 方式 | 内容 | 結果 |
|------|------|------|
| ブラウザXHR(改) | 正しい input にセット→「登録する」クリック→file_upload() を完走 | `doUpload` が **net::ERR_ABORTED**（jQuery 35sタイムアウトabort）。サーバが応答しない。 |
| in-memory File | Playwrightのディスクハンドル由来 File を疑い、base64→メモリBlob で送信 | 変わらず ERR_ABORTED |
| worker直接POST | `page.context().request.post` で doUpload へ直接 multipart POST（ブラウザのCookie共有） | **一度 HTTP 200 成功**したこともあるが、基本は **HTTP undefined / 60sタイムアウト**で `direct_post_blocked`（intermittent） |
| 実Chrome(channel:chrome) | Chrome for Testing ではなく OS の実 Google Chrome | **変化なし**（両方で失敗）。Chrome for Testing が原因**ではない**と確定。 |
| ステルス実Chrome | 自動化フラグ(`--enable-automation`/`AutomationControlled`)除去 + `navigator.webdriver` 偽装 + 専用永続プロファイル | それでも `doUpload` が ERR_ABORTED |
| --no-sandbox 除去 | Chromeの「サポートされていないフラグ」警告(=自動化の目印)を消す | 検証中(v0.2.134) |
| ユーザープロファイル seed | 普段使いChromeの Cookies/Local State 等を専用dirにコピーして起動 | 検証中(v0.2.133〜) |

## 5. 原因の可能性（現時点の評価）

### ★最有力: Akamai Bot Manager によるボット検知
- `doUpload` のリクエストCookieに **`_abck` / `bm_sz` / `ak_bmsc` / `bm_mi` / `bm_sv`**（Akamai Bot Manager のCookie）。
- 失敗時の `_abck` が `~-1~-1~-1` で終わる = **Akamaiが「未検証/失敗」と判定している状態**。
- **手動（人間の育ったプロファイル）は通る・自動（新規/ボット判定セッション）は弾かれる**という挙動はAkamaiのbot検知と完全に一致。
- **「スクレイピングで開いたChromeを手動操作してもダメ」** = 操作方法ではなく、**ブラウザの起動状態（セッションがボット判定済み）**が原因、を強く示す。
- intermittent（たまにHTTP200で通る、エステは一度成功）= Akamaiの判定が確率的であることと一致。

### Akamai がボット判定する具体的な手がかり（潰すべき指紋）
1. `navigator.webdriver = true`（Playwright既定）→ 偽装で対応(v0.2.131)
2. `--enable-automation` フラグ → ignoreDefaultArgs で除去(v0.2.131)
3. **`--no-sandbox` フラグ**（スクショで警告確認）→ 除去(v0.2.134)
4. userAgent 偽装（Chrome/127 と実体 Chrome/147 の不一致）→ UA上書き撤去(v0.2.133)
5. **新規プロファイル = Akamai信頼Cookie(`_abck`)が無い** → ユーザープロファイルseedで対応(v0.2.133)
6. CDP接続/その他のPlaywright指紋（未対策、残課題）

### 副次的に効いている可能性
- 直接POST(`page.request`)は**ブラウザのTLS指紋/Akamaiセンサーを持たない**ため、ブラウザXHRよりさらに弾かれやすい → 既定をブラウザXHRに戻した(v0.2.132)。

## 6. 現在の対策スタック（v0.2.135 時点）

1. **送信はブラウザXHR**（本来の `file_upload()`）を既定。worker直接POSTは `SALONBOARD_DIRECT_POST=1` の時だけ。
2. **ステルス実Chrome**で起動: `--enable-automation`/`--no-sandbox`/`AutomationControlled` 除去、`navigator.webdriver` 偽装、UA偽装なし。
3. **ユーザーのChromeプロファイルを seed**（`~/Library/.../Chrome/Default` の Cookies/Local State 等をコピー）→ Akamai信頼Cookieを引き継ぐ。
4. **自動診断ログ**（v0.2.135）: テスト投稿で `webdriver` / UA / Akamai-cookie 有無 / `doUpload` の実HTTPステータス を出力。

環境変数（実PCで切替可）:
- `SALONBOARD_USE_USER_PROFILE=0` … ユーザープロファイルseedを無効化
- `SALONBOARD_CHROME_SOURCE_DIR` / `SALONBOARD_CHROME_SOURCE_PROFILE` … seed元の場所/プロファイル名
- `SALONBOARD_DIRECT_POST=1` … worker直接POSTを使う（切り分け用）
- `SALONBOARD_FORCE_BUNDLED_CHROMIUM=1` … 同梱Chromiumを強制

---

## 7. 次に確認すること / 残る手

1. **v0.2.135 の自動診断ログ**で確定したい:
   - `webdriver=false` か？ `UA=Chrome147` か？ `Akamai-cookie` はあるか？ `doUpload → HTTP ?`（pending/403/429/200）。
2. プロファイル seed が効いているか（「あなたのChromeプロファイルをシード✅」がログに出るか／実PCのログイン用プロファイルが `Default` か）。
3. seed が**古くなる**問題: `.seeded` マーカーで初回のみコピー。手動セッションでAkamai cookieが更新されたら、専用dirを削除して再seedが要る。**自動再seed**の仕組みが要検討。
4. それでも弾かれるなら:
   - **CDP指紋の除去**（playwright-extra stealth 相当のさらなるパッチ）。
   - **人間の操作を待つ運用**: アップロードの瞬間だけ実ブラウザを表示し、Akamaiチャレンジ(reCAPTCHA等)が出たら人間が解く半自動。
   - **アップロードだけ手動・他は自動**のハイブリッド運用。
