# v0.3.0 計画: 実機 Electron を jobs API polling 型に統合

> v0.2.3 までで「credentials / overview / report-error は Admin API 経由」「device 認証必須」「設定 self-check」が揃った。
> v0.3.0 では次の段階として、**実機 Electron を Admin のジョブキュー (salonboard_sync_jobs) と接続**し、push_blog 等の本格稼働を目指す。

## 1. 目的

| 項目 | 現状 (v0.2.3) | v0.3.0 目標 |
|---|---|---|
| 認証 | device token (済) | device token (継続) |
| credential 取得 | Admin API (済) | Admin API (継続) |
| ジョブ起動 | UI 同期ボタン (直叩き) | jobs API polling (job claim) |
| 結果反映 | Supabase 直書き | callback API 経由 |
| push_blog | 未稼働 (flag false) | **稼働** |
| fetch_bookings / sales / push_booking / cancel_booking | UI 直叩き | jobs API claim → callback |
| Supabase 直書き | bulk_upsert RPC 多数 | 徐々に callback API に寄せる |

## 2. 既存 UI 同期ボタンとの共存方針

「同期」ボタンを 1 日で全廃すると現場が混乱するので、**段階的共存**:

- **共存期間** (v0.3.0 リリース時点):
  - 同期ボタンは残す。ただし内部実装を切り替え:
    - 旧: renderer → workerSync → Playwright 直起動
    - 新: renderer → Admin API に **手動 fetch_* ジョブを enqueue** → Worker (jobs API polling) が拾って実行
  - UI からは「同期する」操作は変わらない。裏で job 経由になるだけ
- **完全移行** (v0.4.0 想定):
  - UI 同期ボタンを撤去 (job のスケジュール実行に統一)

## 3. background polling の ON/OFF

新しい設定項目を追加:
```
SALONBOARD_POLL_ENABLED=true|false   # Worker 内蔵 polling を起動するか
SALONBOARD_POLL_INTERVAL_MS=60000    # poll 間隔 (デフォルト 60s)
```

renderer に「自動同期を一時停止」トグルを追加し、polling を即時 ON/OFF できるようにする (現場が「今日は触らないで」と思ったときに止められる)。

## 4. 手動同期ボタンを job enqueue に変えるか

**変える**。理由:
- 直接 Playwright を起動すると同 PC で複数同時起動が起こり得る (shop_lock があっても device 内のリソース競合)
- 履歴が salonboard_sync_jobs に統一されると Admin の管理がシンプル
- 失敗時の retry 戦略を 1 箇所に集約できる

実装:
- 「同期」ボタン → `POST /api/salonboard/jobs/enqueue` (新設) で fetch_bookings / fetch_sales / fetch_staff 等を即時 (run_at = now) で積む
- Worker は polling で拾って実行
- 結果は callback で返す
- UI には進捗イベント (run/shop:start/progress/end) を polling か WS で流す

## 5. worker.ts を統合するか worker-process.cjs に jobs API 対応か

**結論: worker.ts のロジックを worker-process.cjs に移植**する。

理由:
- worker.ts は CLI + Playwright スタンドアロン (Node 主体)
- worker-process.cjs は Electron utilityProcess (Node + IPC、既に storageState / shop_lock / classifyError 実装あり)
- 後者の方が安全装備が厚いので、worker.ts の job 処理ループだけを移植する方が安全

移植する関数:
- `fetchJobs(limit)` (jobs API polling)
- `pushBlog(page, job)` (ブログ投稿)
- callback の status 8 値マップ
- not_implemented の扱い

worker.ts は **dev/CI 用 dry-run** として残置 (`worker:once` で 1 ジョブだけ処理する用途で重宝する)。

## 6. push_blog の有効化順序

慎重に段階公開:

1. **Phase A**: worker-process.cjs に push_blog 実装移植 (まだ false のまま)
2. **Phase B**: 開発組織だけ `SALONBOARD_PUSH_BLOG_ENABLED=true` で試験
   - DRY_RUN で flow 通電確認
   - 1 店舗で実機ブログ投稿成功確認 (担当者立会)
   - external_id が DB に正しく入るか確認
3. **Phase C**: 1 組織 → 全組織へ env 切替
4. **Phase D**: Admin UI で組織別 ON/OFF (組織単位 feature flag) を追加し、env でなく DB で制御

## 7. 既存 fetch_bookings フローを callback API 化するタイミング

push_blog の手前で:
- fetch_bookings の callback ルートは既に存在 (`upsertBookings` 経由)
- 実機を polling 型に切り替えた瞬間に自動的に callback 経由になる
- 移行期は新旧両系統が動くので **同 booking が二重に upsert される懸念あり** → `external_booking_id` の UNIQUE 制約で冪等性は確保済 (確認)

## 8. rollback 方針

- v0.3.0 を release した後で問題が出たら **`SALONBOARD_POLL_ENABLED=false`** を Admin 側で配ることで polling を停止可能
- Worker は polling を切ると **旧 UI 同期ボタンに自動 fallback** する設計にする (= 既存挙動に戻る)
- push_blog の問題は `SALONBOARD_PUSH_BLOG_ENABLED=false` で即座に止められる (Admin .env)
- DB スキーマは v0.2.3 までで完成しているので migration rollback は不要

## 9. feature flag 一覧 (v0.3.0 で追加予定)

| flag | 既定 | 効果 |
|---|---|---|
| `SALONBOARD_POLL_ENABLED` | true | Worker 内蔵 jobs API polling の起動 |
| `SALONBOARD_POLL_INTERVAL_MS` | 60000 | polling 間隔 (ms) |
| `SALONBOARD_PUSH_BLOG_ENABLED` (既存) | false → true で段階移行 | push_blog enqueue 許可 |
| `SALONBOARD_JOB_ENQUEUE_VIA_UI` | true | 「同期」ボタンを job enqueue 経由にする |
| `SALONBOARD_LEGACY_UI_SYNC_FALLBACK` | true | jobs API 不調時に旧 UI 同期に自動 fallback |
| `SALONBOARD_DISABLE_DIRECT_RPC` | false → true で完全移行 | renderer/worker から salonboard_credentials* の直接 RPC を全面禁止 |

## 10. テスト項目 (v0.3.0 リリース前必須)

### 認証 / scope
- [ ] central-dev token を本番 Admin で使うと 401 になる
- [ ] revoked device で jobs API claim 不可
- [ ] paused device で jobs API claim 不可
- [ ] 未紐付 shop の job は claim されない
- [ ] blocked_until 中の shop は claim されない
- [ ] consent 無し shop は claim されない

### push_blog
- [ ] flag=false の間は enqueue されない & 既存 queued が cancelled に整理される
- [ ] flag=true で公開 → 5分後に job が走り、external_id が `content_posts.salonboard_external_id` に入る
- [ ] external_id 不明時に **仮 ID が入らない** (= 重複投稿事故防止)
- [ ] DRY_RUN モードでは実投稿せず succeed + last_error に注意書き

### fetch_bookings 互換
- [ ] 同 booking が二重に upsert されない (external_booking_id UNIQUE)
- [ ] 旧 UI 同期と新 polling が並走しても DB 不整合なし

### 状態反映 / UI
- [ ] retryable_failed は指数バックオフで再 queue
- [ ] captcha_detected / blocked / login_required / rate_limited は自動 retry されない
- [ ] not_implemented job は succeeded で記録されない
- [ ] /admin/salonboard/devices の連携状態カードに sync_status / blocked_until / last_error が出る

### rollback
- [ ] `SALONBOARD_POLL_ENABLED=false` にすると Worker が polling 停止
- [ ] fallback で旧 UI 同期に戻る
- [ ] `SALONBOARD_PUSH_BLOG_ENABLED=false` に戻すと push_blog enqueue が停止 + 既存 queued が再度 cancelled へ

## 11. v0.3.0 マイグレーション

おそらく不要 (v0.2.x で DB は完成済)。
ただし以下があれば追加:

- `salonboard_sync_logs` に `error_code` 列 (現在は last_error に prefix する形式)

## 12. 残課題 (v0.3.0 後)

- worker-process.cjs から残る `salonboard_credentials.from()` の最後の 3 箇所を完全廃止
- `salonboard_run_*` RPC を callback API に集約
- `salonboard_credentials_overview` の renderer 直読み (Settings 画面) を維持するか、admin 専用 API に隔離するか
- shop ごとの sync_interval_minutes を device overview API でも返せるようにする
